## Relevant artefacts

### Network-state framing — MTD-driven configuration transitions as the metric substrate

**Source locator:** §1 (Introduction, paragraph after footnote 1); §4.1 (Network states).

**Paraphrase:** Hong et al. define a *network state* as a particular configuration of the network — including hosts, services, OS variants, vulnerabilities, and topology — that changes when an MTD technique is triggered. Other configuration-changing events (system updates, policy changes) are explicitly excluded from the scope: the metrics are designed to attribute change to MTD activity only. Each consecutive pair of states becomes the unit over which the dynamic security metrics are computed; sequential pairs are then aggregated and normalised across the observed state trajectory $S$. This framing is what distinguishes Hong's metrics from the static-snapshot metric tradition (e.g., attack-graph length, exploit-probability product) — the metric value is a property of a *transition*, not of a single configuration.

**Quote (if essential):**
> "the term _network state_ refers to the particular configuration of the network, which may be changed to a different one when MTD techniques are triggered" (§1, footnote 1 of source).

**Maps to:** [`../specs/architecture.md`](../specs/architecture.md) §(g) (L4 evaluation idiom; "comparison across configurations inside this substrate is valid"); [`../specs/architecture.md`](../specs/architecture.md) §(k) item 4 (within-substrate comparability); [`../specs/metrics_semantics.md`](../specs/metrics_semantics.md) §(d) (the comparability boundary inherits Hong's framing — *deltas across consecutive runs/states are the comparable quantity, not absolute magnitudes*).

**Disposition for this thesis:** **adopted-as-baseline (framing).** The "compute across consecutive network states" stance is the same one MTDSim's metrics already adopt at evaluation time (per-checkpoint deltas in `evaluation_result_by_compromise_checkpoint`); this paper supplies the published anchor for that framing in the lit review's Table I (*†* footnote). No implementation work needed — the substrate already produces per-state slices that admit Hong-style differencing.

### Attacker-side path-based dynamic metrics — APV, APN, APE

**Source locator:** §5.1.1 (APV, Eq 1–2); §5.1.2 (APN, Eq 3–5); §5.1.3 (APE, Eq 6); §5.3.1–5.3.3 (worked computations on the example SDN, $S = \{s_0, s_1, s_2, s_3\}$, $|S|=4$).

**Paraphrase:** Three attacker-side, reconnaissance-class metrics are defined over consecutive network states:

- **APV (Attack Path Variation)** measures the proportion of attack paths that *change* between $s_{i-1}$ and $s_i$. For each consecutive pair, the per-pair quantity is the cardinality of the set difference $|AP_i \setminus AP_{i-1}|$ normalised by $|AP_i|$; the overall APV averages this across the $|S|-1$ pairs. A low APV means the attack-path set is largely static across MTD activity — i.e., the MTD is not invalidating reconnaissance. The worked example yields APV $= 0.6667$ on the four-state SDN scenario.
- **APN (Attack Path Number)** captures the *proportion change in cardinality* (not membership) of the attack-path set across the same consecutive-pair structure. The per-pair quantity reduces to 1 when the count is unchanged and trends toward 0 when the previous state had no attack paths; the metric is normalised over the $|S|-1$ pairs. The example yields APN $= 1$ because all four states share the same path count of five.
- **APE (Attack Path Exposure)** captures the *duration* each attack path is present, using a time function $t(ap_i)$; the metric is normalised by both the number of attack paths and the total number of network states. APE tends toward $1 - 1/|S|$ when each path appears in only one state (best case); it tends toward 0 when paths are present continuously.

The three together cover *which* paths change, *how many* there are, and *how long* each persists.

**Quote (if essential):**
> "if the previous network state had no attack paths, then it equates to 0" (§5.1.2 of source, in the APN-bound discussion).

**Maps to:** [`../specs/mtdsim_spec.md`](../specs/mtdsim_spec.md) MET-05 (Attack Path Exposure — Ho's `V_new(h)` formulation traces the same "path exposure across states" lineage, though the per-host new-vuln-fraction Ho computes is a different operationalisation of the same idea); [`../specs/mtdsim_spec.md`](../specs/mtdsim_spec.md) MET-09 (Shortest Attack Path Variability — Ho's SAPV is a same-shape, narrower instance of APV restricted to the *shortest* path set rather than the full path set); [`../specs/mtdsim_spec.md`](../specs/mtdsim_spec.md) MET-13 (Dynamic security metrics — generic row noting APV/NAP/ESP/AI as unimplemented at named-output granularity, traces to this paper through Ho's literature review).

**Disposition for this thesis:** **partially adopted as baseline; substrate-divergent on naming and granularity.**
- **APE** — adopted in spirit via MET-05, but the substrate implementation (Ho-derived "new vulns / hosts on the shortest attack path") is not Hong's path-duration formulation. The two metrics share a name and a conceptual lineage but compute different quantities; flag for §(j) when discussing the methodological inheritance chain.
- **APV** — not implemented at the *full*-attack-path-set granularity; the substrate implements MET-09 (SAPV) which is APV restricted to the shortest-path subset (Ho §3.3.2 #10). This is a divergence in scope, not in form.
- **APN** — no substrate counterpart at this name. The information is recoverable from `network.scorer.shortest_path_record` slices but not materialised as a named output.

None of the three is an L4 evaluation-priority metric for this dissertation (architecture §(g) puts internal MTTC as the primary discriminator, with ASR / RoA / attack-path-exposure as secondary). Hong's importance here is *framing-level* — providing the published anchor for the "consecutive states" idiom — not implementation-level.

### Defender-side cost-class metrics — NVC, EVC (and timing complements NVDT, EVT)

**Source locator:** §5.2.1 (NVC, Eq 10–12); §5.2.3 (EVC, Eq 15); §5.2.2 (NVDT, downtime complement to NVC, Eq 13–14); §5.2.4 (EVT, edge-variation time, Eq 16–17); §5.3.6–5.3.9 (worked computations).

**Paraphrase:** The defender-side dynamic metrics in this family capture the *cost* the MTD imposes — its "efficiency" axis in the Cho et al. partition. Two cost dimensions are formalised:

- **NVC (Node Variant Cost)** is the per-state cost of assigning chosen variants (OS, software version, etc.) to hosts, normalised by the maximum allowable variant cost per state and aggregated across $|S|$ states. **NVDT (Node Variant Downtime)** is the parallel time-cost: downtime incurred when assigning new variants, normalised by maximum downtime.
- **EVC (Edge Variation Cost)** counts the number of edge changes (per-state, normalised by the maximum edge-variation cost) and aggregates across the trajectory. **EVT (Edge Variation Time)** is the analogous *time* aggregate, normalised by the maximum amount of time taken for edge-set changes.

Both pairs (NVC/NVDT, EVC/EVT) trend toward 1 when cost is low and toward 0 when cost approaches the per-state maximum. Hong notes that these are *modular*: any subset can be combined into a compound metric via a weighted sum (Eq 27), with weights summing to 1 — the example uses uniform 1/3 weights to produce a single defender-side cost summary across NVC, EVC, and (separately) NVDT, EVT.

**Quote (if essential):**
> "for defense efforts, a metric value toward zero is making the defense more difficult, vice versa" (§6.1.1 of source).

**Maps to:** [`../specs/mtdsim_spec.md`](../specs/mtdsim_spec.md) MET-13 (Dynamic security metrics — generic row); no direct named-output row for NVC or EVC in the current substrate (substrate exposes RoA / ASR / attack-path-exposure / HCR on the L4 face; the cost axis Hong formalises is not surfaced at metric granularity).

**Disposition for this thesis:** **contrasted-against.** The dissertation's L4 evaluation reports the attacker-side discriminators (MTTC, ASR) and is silent on defender-cost metrics by scope: the defence side is deliberately frozen at "existing mechanisms only" (architecture §(a) decision block). Hong's NVC / EVC framework is the published reference for *how* a cost-axis evaluation would be conducted if it were in scope; flagging here so the omission is principled rather than incidental. If the §(l) open-question on the L4 evaluation matrix shape later resurrects cost-axis reporting, Hong's NVC / EVC / NVDT / EVT formulation is the place to start.

### T-HARM substrate and compound-metric construction

**Source locator:** §4 (Temporal HARM); §4.2 Definitions 1–4; §4.3 Definitions 5–8 (MTD-induced changes per Shuffle / Diversity / Redundancy); §5.3.10 (compound metrics, Eq 27).

**Paraphrase:** Hong et al. build the metric machinery on a **Temporal Hierarchical Attack Representation Model (T-HARM)**: a sequence $N = \{n_{t_0}, n_{t_1}, \ldots, n_{t_n}\}$ of two-layered HARMs (upper layer: Attack Graph capturing host reachability; lower layer: per-host Attack Trees capturing vulnerability structure), with each $n_{t_i}$ tied to a network state $s_{t_i}$. MTD techniques are categorised as Shuffle, Diversity, or Redundancy, and each category is given a formal definition of *what* it changes in the T-HARM: Shuffle alters the upper-layer edge set ($E_{t_{i+1}} = E_{t_i} \triangle \{(h_x, h_y)\}$); Diversity alters the lower-layer vulnerability set ($A_{t_{i+1}} = A_{t_i} - d(h_{t_{i+1}})$); Redundancy adds replicated nodes/vulns with the corresponding edge/Attack-Tree consequences. The SDR partition Hong inherits from Hong & Kim 2016 is thus given a representation-level realisation here: the *structural* effect of each MTD class is named precisely on the model.

The compound-metric construction (Eq 27) is a weighted linear combination of any subset of the named modules: $M_{\text{compound}} = \sum w_i m_i$ with $\sum w_i = 1$. Hong notes that weight assignment is itself application-dependent — a hospital network would prioritise reconnaissance-class metrics, a media firm would prioritise availability-class — and that systematic weight derivation is out of scope.

**Maps to:** [`../specs/architecture.md`](../specs/architecture.md) §(h) glossary (the SDR-family definition cross-references Hong 2018 alongside Cho 2020); [`../specs/architecture.md`](../specs/architecture.md) §(i) substrate-seam map (the inherited HARM-structural + DES-execution composition decision block notes Cho's four-method composition; Hong's T-HARM is one published exemplar of a graphical-security-model substrate). MTDSim itself uses a single-time HARM (no temporal sequence variable on the structural side; the DES engine carries time externally), so the T-HARM construction is a published reference point rather than an inherited construct.

**Disposition for this thesis:** **contrasted-against (substrate)** / **adopted-as-baseline (SDR partition)**. MTDSim's substrate is HARM-structural + SimPy-DES, with time advanced by the simulator rather than indexed into the structural model itself; the T-HARM construction is a closely-related but distinct substrate choice (the temporal index is inside the model in T-HARM). The SDR partition Hong formalises *is* the partition this dissertation inherits via Cho 2020 and the lineage four. The compound-metric weighted-sum mechanism is not in scope — internal MTTC is reported standalone per architecture §(g), and no compound score is on the L4 menu — but the construction is named here so any future scope expansion has the reference.

## Open questions / things to verify

- **Hong's "Attack Path Exposure" (path duration) vs Ho's "Attack Path Exposure Score" (per-host new-vuln fraction on the shortest path):** these share a name and lineage but compute different quantities. Worth a sentence in §(j) when discussing how the metric-name terminology flows from Hong → Ho → this substrate.
- **APV vs SAPV scope:** the substrate exposes SAPV (shortest-path subset, MET-09) but not APV at the full path-set granularity. If a finding ever depends on the full-set APV, it would need to be materialised from `network.scorer` — flag as a possible extension point, not as a current need.
